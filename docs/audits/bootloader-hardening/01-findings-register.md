## Feature-gate proof (cfg-trace verdicts)

Feature graph (Cargo.toml): `default=["logging","zk-groth16","zk-vk-provisioned","zk-zeroize"]`, `hardened=["standard"]`, `production=["hardened"]`, `hardened-production=["production"]`, `dev-qemu=["dev-mode"]`, `zk-zeroize=["zeroize"]`, `zk-snark=["zk-groth16"]`. `standard=[]`, `dev-mode=[]`, `mock-proof=[]`, `zk-vk-provisioned=[]` are leaf flags. The only `dev-mode` cfg is entry/dev.rs; the only `mock-proof` references are the Cargo.toml declaration (zero source `cfg`).

- **(a) `production`/`hardened-production` cannot transitively enable `mock-proof`.** PROVEN-safe. Transitive closure of `hardened-production ŌåÆ production ŌåÆ hardened ŌåÆ standard` adds only `standard`; none of `hardened`, `production`, `hardened-production`, `standard` lists `mock-proof`. Additionally `mock-proof` gates zero code (W3-08), so even an explicit `--features mock-proof` is inert. Verdict: PROVEN-safe.
- **(b) They cannot skip signature verification.** PROVEN-safe at the compile-feature level, with an S1 runtime caveat. No `cfg(feature=...)` removes or short-circuits `verify_signature_bytes` / the `verify_signature` enforcement; signature enforcement is governed by the runtime `SecurityMode::requires_signature()` (mode.rs:28), not any feature. A `production` build still compiles the full Ed25519 path. CAVEAT (NOT a feature-gate violation, but in-scope per task wording "any path where a production build can weaken/skip a trust check is S1"): the ungated `Development` menu entry (W3-01) lets a console attacker pick a mode whose `requires_signature()` is false on a production binary. Verdict: PROVEN-safe (no cfg skips sig) + S1 runtime bypass W3-01.
- **(c) They cannot skip ZK verification.** PROVEN-safe at the compile-feature level, fail-closed when `zk-groth16` is absent. With `zk-groth16` (in `default`, and pulled by `zk-snark`) the Groth16 backend runs (zk/verify/core.rs:55-69). Under `--no-default-features --features hardened-production` (no `zk-groth16`), `verify_backend` is the `#[cfg(not(feature="zk-groth16"))]` arm (core.rs:71-74) returning `ZkVerifyResult::Unsupported` ŌåÆ `build_result` sets `zk_verified=false` (zk/attest/verify/boot.rs:49) ŌåÆ orchestrate.rs:33 `handle_verification_failed` ŌåÆ `fatal_reset` for non-Development. So a ZK-proof-bearing kernel CANNOT pass without the backend; ZK is fail-closed. The only skip is Development (W3-05), already counted in W3-01. Verdict: PROVEN-safe (fail-closed); residual skip = W3-01/W3-05.
- **(d) `dev-mode`/`mock-proof` code paths are actually excluded under production (not merely named).** PROVEN for `dev-mode` F12 path and `mock-proof`; S1 residual for the menu. `dev-mode`: only consumer is `dev_override`; `#[cfg(not(feature="dev-mode"))]` stub returns `false` (entry/dev.rs:35-38, W3-12) so F12 is excluded. `mock-proof`: no consumers (W3-08). HOWEVER `SecurityMode::Development` itself is NOT `dev-mode`-gated ŌĆö it is an ordinary enum variant offered unconditionally in `DEFAULT_ENTRIES` (W3-01). So "dev paths excluded" holds for the feature-gated ones but the Development *mode* leaks into production via the menu. Verdict: PROVEN for feature-gated dev/mock code; **S1 VIOLATION of the intent** via ungated menu entry (W3-01).

## UNVERIFIED (W3)

- **UNVERIFIED-1 ŌĆö VK-provisioning under non-default production profiles.** `zk-vk-provisioned` is in `default` but NOT pulled by `production`/`hardened-production` (Cargo.toml). A release artifact built `cargo build --release --no-default-features --features hardened-production` would compile WITHOUT `zk-groth16` and WITHOUT `zk-vk-provisioned`, so (i) the build-time VK-size assertion (validate.rs:21, W3-10) is dropped and (ii) ZK is the fail-closed `Unsupported` stub (core.rs:71). Whether production is *intended* to be built with `--no-default-features` or with `default` features ON (keeping `zk-groth16`+`zk-vk-provisioned`) is indeterminate from source alone. Settling experiment: inspect the release build invocation (CI/Makefile/`xtask`/packaging script) ŌĆö `grep -rn 'no-default-features\|--features' Makefile* xtask* .github/ scripts/ 2>/dev/null` ŌĆö and/or `cargo build --release --no-default-features --features hardened-production` then `nm -C target/.../nonos_boot | grep -i groth16` to confirm whether the Groth16 verifier symbol is present in the shipped binary. If production is built with default features retained, (c)/UNVERIFIED-1 are fully PROVEN-safe; if built `--no-default-features`, production ships a ZK-stub binary (still fail-closed for proof-bearing kernels, but no real ZK verification capability) ŌĆö escalate to S1/S2 in Task 8.
- **UNVERIFIED-2 ŌĆö `cfg!(debug_assertions)` resolution for shipped artifacts (W3-09).** `SecurityPolicy::from_build()` returns `Development` only under `debug_assertions` AND no `standard`/`hardened`. Release profile disables `debug_assertions`, so a release no-profile build yields `Standard`, not `Development`. This is determinate by Rust profile semantics, but the *shipped* profile (release vs debug) for production images is a build-system fact not in source. Settling experiment: confirm the release packaging uses `--release` (or a profile with `debug-assertions = false`) via the same CI/Makefile inspection as UNVERIFIED-1.

## W4 ŌĆö Optimization audit (redundant hashing/copying, post-EBS alloc, re-enumeration, blocking delays)

W4 scopes the LIVE boot spine only (same spine as W3; dead trees `tpm/`,`verify/`,`security/verify/`,`crypto/keyring/`,`crypto/keystore_v2/` excluded). Cost-only/cleanliness = S4; post-EBS / no-reclaim-window hazards Ōēź S2. Findings are diagnostic; no code modified. Redundant-hashing verdict: the kernel image is BLAKE3-hashed exactly **once** on the live spine (`kernel_verify/hash.rs:32`); Ed25519's internal SHA-512 over the same bytes (`crypto/verify/bytes.rs:33`) is inherent to the scheme, not a removable redundancy; `loader/security/hash.rs` blake3 is dead (tests/re-export only) so there is no second full-image BLAKE3. The headline cost is therefore not double-hashing but a redundant full-image **memcpy + page allocation** in the file reader (W4-01). Post-EBS allocation verdict: **none** ŌĆö the path after `st.exit_boot_services()` (orchestrate.rs:87) does zero allocations; `copy_memory_map` writes into the pre-EBS-allocated raw `mmap_addr` (W4-08, recorded as a clean negative).
