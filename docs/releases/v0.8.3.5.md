# NØNOS v0.8.3.5 Release Notes

This release represents a significant milestone in making NØNOS a self-sufficient operating system capable of developing itself. The highlight of this version is the introduction of a fully functional Git client that works natively within NØNOS, allowing developers to clone repositories directly from GitHub and work on code without leaving the operating system.

## Native Git Implementation

NØNOS now ships with a complete Git implementation written entirely in Rust and designed specifically for the RAMFS filesystem. The implementation supports cloning repositories from GitHub, pulling updates, creating commits, managing branches, and viewing history. When you clone a repository, NØNOS initializes a local Git structure and configures the remote automatically. The pull command fetches the repository tree from the GitHub API and downloads each file individually, storing them in the RAMFS filesystem where they can be edited using the built-in text editor.

The Git commands follow familiar conventions that any developer would expect. Running git clone with a GitHub URL creates a new directory and sets up the remote tracking. After navigating into the repository directory, git pull fetches all files from the remote branch. The system displays progress as files are downloaded, showing how many files have been received so far. Local operations like git status, git add, git commit, and git log work entirely offline against the RAMFS storage.

## Network Stack Improvements

The network stack has been significantly optimized to maintain system responsiveness during network operations. Previous versions could experience cursor freezing and input lag while waiting for network responses, particularly during TLS handshakes and large file downloads. This release introduces cooperative yielding throughout the network code path, ensuring that the system remains interactive even during heavy network activity.

The TLS handshake implementation now yields periodically during the connection establishment phase, allowing mouse movement and keyboard input to be processed between network round trips. TCP connection establishment and data reception have been similarly updated with strategic yield points that balance network performance with system responsiveness. The HTTP client benefits from these improvements, making browser navigation and Git operations feel much smoother.

## Desktop Environment Enhancements

The desktop environment now supports proper folder navigation. Users can double-click folders on the desktop to navigate into them, and a new Go Back option in the context menu allows returning to the parent directory. Files and folders created while inside a subdirectory are now correctly placed within that directory rather than at the desktop root. This change makes the desktop behave more like a traditional file manager while maintaining its simplicity.

The text editor received a critical fix for the Enter key, which was not creating new lines in certain situations. This issue has been resolved, and the editor now handles line breaks correctly in all cases.

## Technical Details

The Git implementation stores repository data within the RAMFS filesystem under the current working directory. Each repository maintains a standard Git structure with a .git directory containing configuration, refs, and object storage. The system currently supports public GitHub repositories over HTTPS, with authentication for private repositories planned for a future release.

Network operations now include proper progress feedback. When pulling a large repository, the terminal displays status updates as files are downloaded, providing visibility into the operation's progress. The system yields to the event loop every few files, preventing the interface from becoming unresponsive during bulk downloads.

## Looking Forward

This release lays the groundwork for a fully self-hosted development environment. Future versions will expand the Git implementation with support for pushing changes, handling merge conflicts, and working with private repositories through token authentication. The goal is to make NØNOS capable of cloning its own kernel repository, making modifications, and contributing changes back upstream.
