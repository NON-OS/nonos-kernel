  ---
  Most Likely Root Cause: Segment Registers Set to Zero

  The bootloader's jump_to_kernel function at nonos-bootloader/src/handoff/jump/transfer.rs line 29 contains:

  xor edx, edx
  mov ds, dx
  mov es, dx
  mov fs, dx
  mov gs, dx
  mov ss, dx
  This sets SS (Stack Segment) to zero before jumping to the kernel. In 64-bit long mode, most segment register operations are ignored for addressing purposes, but SS=0 has platform-specific behavior.

  The kernel entry function kernel_entry at src/nonos_main.rs line 52 is NOT a naked function. When control reaches kernel_entry, the compiler-generated prologue will execute:

  push rbp        ; This requires SS to be valid
  mov rbp, rsp
  If the processor faults on the first PUSH instruction due to SS=0, the kernel hangs immediately with no output. This happens BEFORE serial::init() is called, which is why no serial output appears.

  Different processors handle SS=0 differently:
  - Intel processors generally tolerate SS=0 in 64-bit mode
  - AMD processors may behave differently
  - Virtualized environments vary in their handling
  - QEMU typically tolerates it, but not always

  The fix is to remove the segment register zeroing or set SS to a valid selector (0x10 matches typical UEFI data segment)

  ---
  Second Likely Cause: Stack Memory Not Accessible

  The kernel stack is allocated at nonos-bootloader/src/handoff/prepare/alloc.rs line 31:

  let stack_addr = bs.allocate_pages(AllocateType::AnyPages, MemoryType::LOADER_DATA, stack_pages)
  The memory type LOADER_DATA should remain accessible after ExitBootServices. However, if there is any bug in the memory map finalization or if UEFI firmware marks this memory differently, the stack push will fault.

  After ExitBootServices at nonos-bootloader/src/handoff/exit/orchestrate.rs line 39, the kernel has no UEFI services and no way to recover from memory access faults. A fault at this stage causes an immediate hang.

  To verify this, we need to examine the memory map passed to the kernel and confirm the stack address falls within a usable memory region.

  ---
  Third Possibility: Entry Point Calculation Error

  The kernel entry point is calculated at nonos-bootloader/src/loader/core/dynamic/entry_validate.rs line 23:

  let offset = e_entry.checked_sub(virt_base)
  let entry = phys_base + offset
  For a PIE kernel with linker script starting at address 0 (linker.ld line 17), virt_base should be 0, making entry = phys_base + e_entry.

  If any part of this calculation is wrong, the bootloader jumps to an invalid address. The validate_entry_address function at nonos-bootloader/src/handoff/jump/validate_entry.rs performs sanity checks
  But if the entry point is valid-looking but wrong, the kernel will crash.

  The bootloader logs "OK jumping 0x..." which shows the calculated address. Verify this address against objdump output of the kernel binary to confirm the entry point is correct.

  ---
  How to Diagnose

  Add raw serial output at the very first instruction of _start, before the jmp to kernel_entry. This confirms whether the bootloader transfer succeeded.

  Modify src/nonos_main.rs _start function:

  #[unsafe(naked)]
  #[no_mangle]
  #[link_section = ".text._start"]
  pub extern "C" fn _start() -> ! {
      naked_asm!(
          "mov dx, 0x3F8",     // COM1 data port
          "mov al, 0x4B",      // 'K'
          "out dx, al",
          "mov al, 0x0D",      // CR
          "out dx, al",
          "mov al, 0x0A",      // LF
          "out dx, al",
          "jmp {entry}",
          entry = sym kernel_entry,
      )
  }
  If "K" appears in serial output:
  - The bootloader transfer succeeded
  - Problem is in kernel_entry or later
  - SS=0 may be causing fault on first prologue instruction

  If "K" does NOT appear:
  - The bootloader jump itself failed
  - Entry point calculation is wrong
  - Or processor faulted immediately at _start

  ---
  Code Model Issue: Secondary Concern

  The kernel uses code-model "small" (x86_64-nonos.json line 31) while pqclean libraries use -mcmodel=large (build.rs lines 100 and 173). This IS a bug that will cause crashes when crypto functions are called, but it is NOT the primary boot blocker because:

  1. The boot sequence is: _start -> kernel_entry -> init_core_systems -> serial::init
  2. Crypto is not called until microkernel_init (src/kernel_core/init/entry.rs lines 40-41)
  3. Serial output should appear before crypto initialization

  This must still be fixed for production, but it explains crashes during crypto operations, not the initial boot hang.

  ---
  Additional Issues Found

  These do not block boot but should be fixed for production:

  Stack allocation subtracts 8 instead of 16 at nonos-bootloader/src/handoff/prepare/alloc.rs line 32. The subsequent alignment mask corrects this, but the code is confusing.

  WBINVD instruction at transfer.rs line 26 flushes all caches before kernel jump. This is extremely slow (tens of milliseconds on modern CPUs) and unnecessary. Remove it.

  Dynamic section parsing at nonos-bootloader/src/loader/reloc/types.rs line 97 has no iteration limit. A malformed ELF could cause infinite loop.

  Segment alignment violations at nonos-bootloader/src/loader/core/exec.rs line 95 only warn instead of error.

  Handoff alignment at src/boot/handoff/api/init.rs line 24 uses hardcoded value instead of compile-time constant.

  ---
  What Was Verified Correct

  Kernel entry point sequence: The naked _start with jmp to kernel_entry correctly preserves RDI (handoff pointer). The jmp instruction does not modify registers, so System V ABI parameter passing is correct.

  PIE relocation: The bootloader correctly applies R_X86_64_RELATIVE relocations by adding load_bias (phys_base - virt_base) to addends.

  Address validation: Entry, stack, and handoff addresses are validated for non-zero, above 1MB, canonical form, and proper alignment before the jump.

  GDT structure: The kernel's GDT at src/sys/gdt/types.rs has correct null, code (0x9A/0xAF), and data (0x92/0xCF) descriptors.

  Global allocator safety: The SecureHeapAllocator returns null before initialization (src/memory/heap/types/alloc_impl.rs line 27-29) and nothing allocates before init_core_systems.

  ---
  Boot Sequence

  1. Bootloader calls exit_and_jump (orchestrate.rs line 29)
  2. Allocates handoff structure, stack (64KB), memory map buffer
  3. Copies memory map after ExitBootServices
  4. Validates entry, stack, handoff addresses
  5. Calls jump_to_kernel with entry=kernel_address, stack=stack_top, handoff=struct_address
  6. jump_to_kernel disables interrupts, flushes caches
  7. Sets segment registers to 0 (THIS IS THE LIKELY BUG)  <---
  8. Sets RSP to stack address
  9. Zeroes general purpose registers except RAX (entry), RDI (handoff)
  10. Executes jmp rax to transfer to kernel

  If step 7 (SS=0) causes processor-specific behavior, or step 8 points to invalid memory, the first instruction in _start or kernel_entry will fault.

  The _start function does only "jmp kernel_entry" which does not touch the stack. But kernel_entry is a normal Rust function with prologue:

  push rbp        ; Requires valid stack
  mov rbp, rsp
  This PUSH is the first stack operation. If it faults, nothing runs.

  Conclusion

  The kernel boot failure is most likely caused by SS=0 before kernel entry. The first stack operation (push rbp in kernel_entry prologue) may fault on certain processor implementations or virtualization environments.

  ## To note ALSO is the code model mismatch between kernel and pqclean is a real bug that will cause crashes during crypto operations but it is not the initial boot blocker because crypto is not called until after serial output should appear.

  To confirm the diagnosis, add raw serial output in _start before any stack operations. If output appears, SS=0 is not the issue and investigation should focus on kernel_entry prologue and early initialization. If no output appears, the bootloader jump itself is failing.

  After fixing the segment register issue, the kernel should boot far enough to show serial output. Then fix the code model mismatch to prevent crypto crashes.
