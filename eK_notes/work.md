# What's actually been done

Phase 1 containment plus a partial Wave 2 deletion cascade. The longer version is below.

The containment pass froze thirteen subsystem trees. Each had a clear canonical winner named by the audit and a clear loser. Each time the pattern was the same. A freeze comment in plain prose at the top of the loser's `mod.rs`. Public exports narrowed down to whatever live callers actually needed. A paired canonical-owner annotation on the winner. About 133 vestigial public names came off the surface that way. Two `mod.rs` files still had implementation logic in them; that logic got extracted into `src/sched/cpu_stats.rs` and `src/memory/api.rs` so the mod files became exports-only. The audit's most concrete trust-critical findings got recorded directly in source where they belong. The literal zero-key Ed25519 verifier in `nonos_module_loader/manager.rs:63`. The std-build verification bypass at line 57. The NOP-sled `verify_and_queue` at lines 176 to 200. Wave 6 will not have to rediscover them.

Several of the freezes uncovered duplicate trees the audit had not named. `src/sys/process` turned out to be a third scheduler universe with its own `Task`, `TaskState`, `SchedulerPolicy` and an 800-line test file. `src/modules/nonos_auth` and `src/modules/nonos_manifest` were hidden parallels of canonical auth and manifest. All three got frozen in the same wave they were discovered.

The Wave 2 deletions removed the frozen trees that turned out to be load-bearing for nothing. Six `nonos_*` module trees came out in a four-step cascade ordered by dependencies. Deleting `nonos_loader` immediately made `nonos_auth`, `nonos_registry` and `nonos_mod_runner` zero-consumer, which made `nonos_sandbox` and `nonos_manifest` zero-consumer in the next pass. `src/drivers/tty` was filesystem-orphan code that was never even declared as a module, so it just got deleted. `src/ui` was effectively dead. The whole tree gated behind `feature = "ui"`, holding a `cli::spawn()` that did nothing but register two no-op commands, plus a clipboard duplicating `src/graphics/clipboard`. The whole tree came out, the surface migrations were done (clipboard moved to `graphics`, the no-op CLI calls removed from boot) and the `cli` and `ui` features in `Cargo.toml` were dropped along with it. `src/sys/process` was removed once `cmd_top` got rewritten against the canonical process table.

The scheduler work was the first real architectural lift, not just freezes and deletions. The audit said scheduling authority belongs under `src/process/scheduler` but the live dispatcher was in `src/sched`. Rather than try to move it all at once, four pieces moved as separate clean cuts on the same branch, each its own commit. The PID run queue and sleep table (`dispatch`) went first, because the files already spoke to the canonical `PROCESS_TABLE` and only the namespace was wrong. Then the preemption atomics and mechanics: `SCHEDULER_STATS`, `NEED_RESCHEDULE`, `CURRENT_TIME_SLICE`, plus `tick`, `switch` and `yield_impl`. Lifting them closed the cross-tree dependency the dispatch lift had opened. Task selection went next: `select_next_process`, `switch_to_process`, `LAST_SCHEDULED_PID`. Then the scheduler types: `Scheduler`, `SchedulerStats`, `SchedulerStatsSnapshot`. Git detected each lift as a rename so the history reads as a relocation, not a deletion-and-recreate.

The repo is materially different now from where it started. Two whole subsystem directories that were on disk a week ago are not anymore. The scheduler authority the constitution says belongs under process lifecycle actually lives there for the most concrete pieces. Legacy `crate::sched::*` paths still resolve through forwarding aliases. That is deliberate. The goal was not to break callers. The implementation has moved. What remains in `src/sched/scheduler/` is `core.rs` (the dispatcher loop), `module_tasks/`, `smp/`, `stats.rs` and the four forwarding aliases. Everything else either lifted out or is one of the lower-level kernel primitives that probably should not move at all.
